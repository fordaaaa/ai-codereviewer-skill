# cr-map — codebase mindmap prompt

Portable version of Claude Code's `/cr-map` command. Paste this into any coding agent. Replace `{{MODE}}` with `refresh` (default) or `section <name>` first.

---

Generate a compact, machine-readable **codebase map** — a mindmap of this repo's architecture and attack surface — and write it to `.claude/cr/codebase-map.md`. The point is to have a terse, stable file an AI agent can re-read later instead of re-exploring the whole tree. Read-only with respect to source code; only write the map file.

Mode: `{{MODE}}` — `refresh` rebuilds the whole map (updating in place if one exists, preserving lines marked `<!-- keep -->`); `section <name>` rebuilds only that section.

## Step 1 — explore

If your tool supports subagents/parallel sessions, fan out; otherwise do sequential focused passes. Gather, without reading every line:
- Top-level modules/services and one line each on purpose.
- Entry points — HTTP routes, CLI commands, event/queue handlers, cron jobs, webhooks (where untrusted input enters).
- Trust boundaries — request handlers, deserialization, auth middleware, template rendering, shell-out points.
- Sensitive sinks — SQL/query builders, exec/spawn/shell, filesystem writes, eval/template compile, crypto, outbound HTTP, secret reads.
- Data stores & integrations, and which env vars configure them.
- Auth model — how authn/authz is enforced and where.
- Build/test/start commands.

## Step 2 — write the map

Write `.claude/cr/codebase-map.md` with this schema (omit empty sections). Every entry-point/sink line ends with a `file:line` anchor.

```
# Codebase Map
<!-- generated by cr-map. Lines marked `<!-- keep -->` are preserved on refresh. -->
_repo: <name>  ·  commit: <short-sha>  ·  updated: <ISO-date>_

## Modules
- <path/> — <one-line purpose>

## Entry points
- <METHOD /route or command> → <handler> (`file:line`) — <authed? input shape?>

## Trust boundaries
- <where untrusted data crosses in> (`file:line`)

## Sensitive sinks
- [sql|exec|fs|crypto|secret|net] <what> (`file:line`)

## Data stores & integrations
- <store/service> — <purpose> — configured by <env var>

## Auth model
- <one-line summary + where enforced>

## Build / run
- build: <cmd>  ·  test: <cmd>  ·  start: <cmd>
```

Keep it under ~200 lines for a typical repo — it's grounding, not documentation. If a section would be huge, list the most security-relevant items and note the count of the rest.

## Step 3 — report

State that the map was written/updated, how many entry points and sinks it captured, and that review passes should read it first as grounding.
