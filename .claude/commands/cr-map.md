---
description: Build or refresh a machine-readable "mindmap" of the repo (architecture + attack surface) that other cr-* commands read as grounding
argument-hint: "[refresh|section <name>]"
---

Generate a compact, machine-readable **codebase map** — a mindmap of the repo's architecture and attack surface that Claude (and every other `cr-*` command) can load instead of re-exploring the whole tree each run. This is read-only with respect to source code; it only writes the map file.

The map lives at **`.claude/cr/codebase-map.md`**. It is designed to be re-read by an LLM, so keep it terse, stable, and structured — bullet facts, not prose.

## Step 0 — parse argument

Argument: `$ARGUMENTS` (default: full build/refresh).

- empty or `refresh` — (re)build the whole map. If a map already exists, diff against the current tree and update it in place, preserving any hand-edited lines marked `<!-- keep -->`.
- `section <name>` — rebuild only one section (e.g. `section sinks`) and leave the rest untouched. Useful after a focused change.

## Step 1 — explore

Spawn 1–3 `Explore` subagents in parallel (single message) sized to the repo. Have them gather, without reading every line:

- **Top-level layout** — modules/packages/services and one line on what each does.
- **Entry points** — HTTP routes, CLI commands, event/queue handlers, cron jobs, public API surface, webhooks. These are where untrusted input enters.
- **Trust boundaries** — where data crosses from untrusted → trusted (request handlers, deserialization, auth middleware, template rendering, shell-out points).
- **Sensitive sinks** — SQL/query builders, `exec`/`spawn`/shell, file-system writes, `eval`/template compile, crypto calls, outbound HTTP, secret/credential reads.
- **Data stores & integrations** — databases, caches, queues, third-party APIs, and which env vars/secrets configure them.
- **Auth model** — how authn/authz is enforced and where the checks live.
- **Build/run** — how the app is built, tested, and started (so `/verify`-style checks know the entry command).

## Step 2 — write the map

Write `.claude/cr/codebase-map.md` with this exact schema (omit a section only if truly empty). Every sink/entry line ends with a `file:line` anchor so downstream commands can jump straight there.

```
# Codebase Map
<!-- generated by /cr-map — regenerate with `/cr-map refresh`. Lines marked `<!-- keep -->` are preserved. -->
_repo: <name>  ·  commit: <short-sha>  ·  updated: <ISO-date>_

## Modules
- <path/> — <one-line purpose>

## Entry points
- <METHOD /route or command> → <handler> (`file:line`) — <trust note: authed? input shape?>

## Trust boundaries
- <where untrusted data crosses in> (`file:line`)

## Sensitive sinks
- [sql] <what> (`file:line`)
- [exec] <what> (`file:line`)
- [fs] <what> (`file:line`)
- [crypto] <what> (`file:line`)
- [secret] <what> (`file:line`)
- [net] <what> (`file:line`)

## Data stores & integrations
- <store/service> — <purpose> — configured by <env var>

## Auth model
- <one-line summary of how authn/authz works and where enforced>

## Build / run
- build: <cmd>  ·  test: <cmd>  ·  start: <cmd>
```

Keep the whole file under ~200 lines for a typical repo — it is grounding, not documentation. If a section would be huge, list the most security-relevant items and note the count of the rest.

## Step 3 — report

Tell the user the map was written/updated, how many entry points and sinks it captured, and note that `/cr-run` and `/cr-sec` will now pick it up automatically. If the repo changed a lot since a prior map, summarize what moved.
